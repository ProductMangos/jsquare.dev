Logging that Data

Feb 1, 2024

We programmers tend to focus all our efforts in building the application and when it's finally complete, we ship it out to the world. Unfortuntely in our drive to push out the application, we tend to place our logging on the backburner. When bugs appear, we'll always wish we had it in place.

What does it mean to log?

The action of storing data to be referenced outside of the application is logging. The actual data to be stored is usually in the categories of:

  • Trace - Level 0
  • Info - Level 10
  • Debug - Level 20
  • Warnings - Level 30
  • Errors - Level 40
  • Critical - Level 50
  • Fatals - Level 60

The level numbers themselves aren't as important as the name of the category but generally speaking... the more important the error - the higher the level.

When logging is written and executed correctly... it's able to collect all of the wanted data throughout the application and store that information in a place that can be easily referenced outside of the application, at any time, regardless of the applications current state (running or not).

This ability allows us the developers to review any data from our logging methods for debug or performance reasons, this is especially important for "downed" applications. If the application isn't running, it can't tell you whats wrong at the time of the error, but the logs that reside outside of it can.

How to start logging?

We're gonna tackle this in a very simple way and that's by attempting to create a logger ourselves! I'm aware that there are many packages across many languages and frameworks that accomplishes this... but there is nothing better to learn then doing it yourself! So at a high level, we know what we want this thing to do... and that's storing information from our application and putting it somewhere for us to reference.

The text file
